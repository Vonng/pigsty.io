PostgreSQL正在吞噬数据库世界

PostgreSQL 并不是一个简单的关系型数据库，而是一个数据管理的抽象框架，具有吞噬整个数据库世界的力量。而这也是正在发生的事情 —— “一切皆用 Postgres” 已经不再是少数精英团队的前沿探索，而是成为了一种进入主流视野的最佳实践。

OLAP 领域迎来踢馆者

在 2016 年的一次数据库沙龙里，我提出了一个观点： 现在 PostgreSQL 生态的一个主要遗憾是，缺少一个足够好的列式存储分析插件来做 OLAP 分析。

以分析领域的权威评测 Clickbench 为例，这里我们使用了一个最有代表性的用例（耗时相对于基准的倍数，越短越好），并在其中标注出了 PostgreSQL 与生态扩展插件的性能表现。原生未经过调优的 PostgreSQL 耗时（x1000），调优后可以达到（x47）。PG 生态中另外三个与分析有些关系的扩展：列存 Hydra（x42），时序扩展 TimescaleDB（x103），以及分布式扩展 Citus（x262）表现也就在这个水平徘徊。

原生的 PostgreSQL 本身提供了很不错的分析能力，应付一般的分析性负载绰绰有余。但在比较大的数据量下跑全量分析的性能，相比专用的实时数仓仍然有些不够看。

，同时，PG生态还有三个与分析有关系的扩展：列存 Hydra（x42），时序扩展 TimescaleDB（x103），以及分布式扩展 Citus（x262）。但与专注于 OLAP 的第一梯队组件：Umbra，ClickHouse，Databend，SelectDB（x3~x4）相比仍然有十几倍的性能差距。

然而最近出现的 ParadeDB 和 DuckDB 的出现改变了这一点！

ParadeDB 提供的 PG 原生扩展 pg_analytics 实现了第二梯队（x10）的性能表现，与第一梯队的 OLAP 数据库只有 3～4 倍的性能差距。比起额外的好处来说：ACID，数据新鲜性，无需 ETL，额外学习成本，维护独立的新服务，（更别提它还提供了 ElasticSearch 质量的全文检索能力），这种性能差距通常是可以接受的。

而 DuckDB （x3.2）则把 OLAP 拔高到了一个全新高度 —— 抛开 Umbra 这种学术类研究数据库的用例，DuckDB 也许是 OLAP 实战性能最快的分析数据库。虽然说它并不是 PG 的扩展插件，但它是一个嵌入式组件，而 DuckDB FDW 以及 pg_quack 这样的项目能让 PostgreSQL 充分利用到 DuckDB 带来的完整分析性能红利！

就在最近， ParadeDB 与 DuckDB 的出现让 PostgreSQL 的分析性能来到了 OLAP 的第一梯队与金字塔尖，弥补了这最后一块短板。

到了 2024 年，在 PostgreSQL 生态里已经有好几个足够好的分析扩展了。算上衍生与兼容的数据库，在 Clickbench 的排行榜上已经占据了一大块地盘。

OLAP 领域迎来踢馆者

主宰分析数据库的新势力

正如 PostgreSQL 正在 OLTP 领域 压倒 MySQL 一样，同样的事情也发生在分析领域。

作为专用的 OLTP 数据库，原生的 MySQL（x3000） 和 MariaDB（x20000） 本身在 OLAP 表现只能用垃圾来形容。虽然未调优的 PostgreSQL 耗时（x1000）也很拉垮，但经过优化后也能达到能用的水准（x47）。

尽管 MariaDB 似乎提供了一个还能看的列存引擎（x70），但 MySQL 生态的 OLAP 常规最佳实践是选用一个额外的专用 OLAP 组件，来解决这个问题，比如兼容 MySQL 线缆协议的 ClickHouse，或 SelectDB，Doris 等。

这些专业组件的分析性能确实是第一梯队（x3~x4），但缺点正如无数专用数据库组件一样：数据冗余、 大量不必要的数据搬运工作、分布式组件之间缺乏一致性、额外的专业技能带来的复杂度成本、学习成本、以及人力成本、 额外的软件许可费用、极其有限的查询语言能力、可编程性、可扩展性、有限的工具链、以及与OLTP 数据库相比更差的数据完整性和可用性。

用各种组件杂凑而成的所谓 “现代数据技术栈”

PostgreSQL 用户同样可以使用这些专用组件，但他们有着更多的选项 —— Hydra, Citus, TimescaleDB 这些基于 PostgreSQL 内核的扩展插件。如果 PostgreSQL 可以直接在原有内核上通过加装扩展，实现****第一梯队的分析性能，却又不引入额外的组件与性能/管理开销，那么会改变整个游戏规则 —— 作为一个真正的 HTAP 数据库，PostgreSQL 会对各类 OLTP / OLAP 数据库构成降维打击。

极简数据技术栈：一切皆用 Postgres 实现！

在数据库诞生伊始，并没有 OLTP 与 OLAP 的区分。OLAP 数据仓库从数据库中“独立”出来，已经是上世纪九十年代时候的事了。主要驱动力在于：传统的 OLTP 数据库难以支撑起分析场景下的查询模式，数据量与性能要求。

然而天下大势，分久必合，合久必分。硬件遵循摩尔定律又发展了三十年，性能翻了几个数量级，成本下降了几个数量级。在 2024 年的当下，x86 单机可以达到 512 核几个TB的内存，单卡 NVMe SSD 可达 64TB，全闪单机柜 2PB ；而 PostgreSQL 的分析扩展插件弥补了软件的缺憾 —— 一套单机 PostgreSQL 的 OLTP / OLAP 性能足以覆盖 99% 以上的场景需求。

基本工作假设的变化，将重新推动数据库世界走向“分久必合”的阶段，让 OLAP与 OLTP 重新走向统一。新的大一统超融合数据库将会出现，而这个数据库，就是 PostgreSQL。

分久必合的大一统数据库

数据库领域有许多“细分领域”：时序数据库，地理空间数据库，文档数据库，搜索数据库，图数据库，向量数据库，消息队列，对象数据库，OLAP 分析也许是其中拆分最早，范围最广的一个子领域。

在 2016 年的一次沙龙聚会中，我提出了一个观点： 现在 PostgreSQL 生态的一个主要遗憾是，缺少一个足够好的分析列式存储插件来做 OLAP。而到了 2024 年，这个问题终于迎来了那个“足够好”的扩展：ParadeDB & DuckDB FDW。

在 PG 生态中，一个 PostGIS 插件，霸占了地理空间事实标准；一个 TimescaleDB 扩展，让一堆“通用”时序数据库尴尬的说不出话来；一个向量扩展 PGVector 插件，更是让整个**专用向量数据库细分领域** 变成笑话。同样的事情已经发生过很多次，我们也会在 OLAP 领域同样见证这一点

PostgreSQL 要替代的可不仅仅是 OLAP 数仓，而是整个数据库世界！

StackOverflow 2023 调研结果，PostgreSQL 三项全能王** **

PostgreSQL 将成为数据库世界的 Linux 内核，吞噬一切 —— 这并不意味者其他数据库一点活路都没有了 —— FreeBSD，Solaris，Unix，还有一些主打冷门小众细分领域的 OS 现在确实还活着，但服务器操作系统内核，确实已经是 Linux 的天下了。

但 PostgreSQL 有何德何能可当此大任？诚然，PostgreSQL 先进，但 Oracle 也先进；PostgreSQL 开源，但 MySQL 也开源。而 PostgreSQL 独一无二的特点，就是其高度的可扩展性，与繁荣的扩展生态。

可扩展性是 PostgreSQL 的灵魂

PostgreSQL 并不是一个简单的关系型数据库，而是一个数据管理的抽象框架，具有囊括一切，吞噬整个数据库世界的力量。而它的核心竞争力（除了开源与先进）来自可扩展性，即基础设施的可复用性与扩展插件的可组合性。

PostgreSQL 允许用户开发功能模块，复用数据库公共基础设施，以最低的成本交付功能。例如，仅有两千行代码的向量数据库扩展 pgvector 与百万行代码的 PostgreSQL 在复杂度上相比可以说微不足道，但正是这“微不足道”的扩展，实现了完整的向量数据类型与索引能力，干翻了几乎所有专用向量数据库。

向量数据库哪家强？

为什么？因为作者不需要操心数据库的通用额外复杂度：事务 ACID，故障恢复，备份PITR，高可用，访问控制，监控，部署，三方生态工具，客户端驱动这些需要成百上千万行代码才能解决好的问题，只需要关注自己所需问题的本质复杂度即可。

可扩展性带来的另一点巨大优势是扩展的可组合性，例如分布式的扩展 Citus 可以将将单机主从数据库集群原地升级改造为透明水平分片的分布式数据库集群。而这个能力是可以与其他功能正交组合的 —— 让 PostGIS 成为了分布式地理数据库，让 PGVector 成为了分布式向量数据库，让 ParadeDB 成为了分布式全文搜索数据库。再比如说提供全文检索能力的 BM25 扩展可以和提供语义模糊检索的 PGVector 扩展组合使用，提供混合检索能力。

更过分的是，扩展插件是可以独立演进的，不需要繁琐的主干合并，联调协作。因此是可以 Scale 的 —— PG 的可扩展性允许无数个团队并行探索数据库前研发展方向，而完全可选的扩展丝毫不会影响主干核心能力的稳定性。

改变游戏规则的 PostgreSQL

PostgreSQL 的出现，改变了数据库领域的游戏规则：任何试图开发“新数据库内核”的团队，都需要经过这道试炼 —— 新的具有实用意义的通用数据库内核几乎不可能诞生了，因为任何单一数据库都无法与所有扩展加持下的 Postgres 在整体实力上相抗衡 —— 包括 Oracle，因为 PostgreSQL 还有开源免费的必杀技。

而某个细分领域的数据库产品，如果能在单点属性（通常是性能）上相比 PostgreSQL 有一个数量级的优势，那也许还能挣得些许生态位苟活一下。但用不了多久，便会在 PostgreSQL 生态扩展的开源替代下被碾压而过。因为选择开发扩展，而不是一个完整数据库的团队会在追赶复刻速度上有碾压性优势！

举个例子，ElasticSearch 基于 Lucene 搜索库开发，而 Rust 生态有一个改进版的下一代 Tantivy 全文搜索库作为 Lucene 的替代；而 ParadeDB 只需要将其封装对接到 PostgreSQL 的接口上，即可提供比肩甚至超越 ElasticSearch 的搜索服务了。更重要的是，它可以站在 PostgreSQL 巨人的肩膀上，借用 PG 生态的全部合力（例如，与 PG Vector 做混合检索）去与一个专用数据库单品来对标。

正如《技术极简主义：一切皆用 Postgres 》所言：简化技术栈、减少组件、加快开发速度、降低风险并提供更多功能特性的方法之一就是 “一切皆用 Postgres”。Postgres 能够取代许多后端技术，包括 MySQL，Kafka、RabbitMQ、ElasticSearch，Mongo和 Redis，至少到数百万用户时都毫无问题。

而现在，我们可以把最后一块拼图 —— OLAP 数仓加入到里面。一切皆用 Postgres ，已经不再是少数精英团队的前沿探索，而是成为了一种进入主流视野的最佳实践。PostgreSQL 正在吞噬整个数据库世界，成为数据库领域的 Linux 内核。而各种开发者调研都在印证着这一点。

还有什么可以做的？

如果按照这样的展开逻辑，我们已经不难预见到数据库领域的终局。但在这个过程中，我们又能做什么，又应该做什么呢？

PostgreSQL 对于绝大多数场景都已经是一个足够完美的数据库内核了，在这个前提下，所谓数据库内核卡脖子纯属无稽之谈。这些 Fork PostgreSQL 和 MySQL 并以内核魔改作为卖点的所谓“数据库” 基本不会有大出息。

—— 这就好比今天我们看 Linux 操作系统内核一样，尽管市面上有这么多的 Linux 操作系统发行版可供选择，大家都不约而同选择使用同样的 Linux 内核，选择吃饱了撑着魔改内核的人，属于没有困难创造困难也要上，会被业内当成山炮看待。

同理，数据库内核本身已经不再是主要矛盾，焦点将会集中到两个方向上 —— 数据库扩展与数据库服务！前者体现为数据库内部的可扩展性， 后者体现为数据库外部的可组合性。当然，也会有去整合扩展与服务的数据库发行版。

例如，MariaDB 作为 MySQL 的亲爹 Fork 已经濒临退市，而靠卖 MySQL RDS 数据库的 AWS 可以赚的钵满盆翻。做 PostgreSQL 内核的厂商不温不火，而投资机构已经出手了许多 PG 生态的扩展插件与服务发行版。例如 ParadeDB 就是 YC 23 刚投的项目。

PostgreSQL 生态中的一个困境就是，许多扩展插件，生态工具都是独立演进，各自为战的，没有一个整合者能将他们凝聚起来形成合力。例如，提供分析的 Hydra 会打一个包一个 Docker 镜像， PostgresML 也会打自己的包和镜像，各家只发行加装了自己扩展的 Postgres 镜像。而这些朴素的镜像与包也距离 RDS 这样完整的数据库服务相距甚远。

即使是类似于 AWS RDS 这样的服务提供商与生态整合者，在诸多扩展面前也依然力有所不逮，只能提供其中的少数。更多的强力扩展出于各种原因（AGPLv3 协议，多租户租赁带来的安全挑战）而无法使用。

许多关键扩展在RDS中并不可用

知行合一的实践

虽然接触 MySQL 和 MSSQL 要早得多，但我在 2015 年第一次上手 PostgreSQL 时，就相信它会是数据库领域的未来了。快十年过去，我也从 PG 的使用者，管理者，变为了贡献者，开发者。

在与形形色色的用户沟通交流中，我发现了数据库领域的木桶短板不是内核 —— 现有的 PostgreSQL 已经足够好了，而是用好数据库内核本身的能力，这也是 RDS 这样的服务赚的钵满盆翻的原因。

但我希望这样的能力，应该像自由软件运动所倡导的理念那样，像 PostgreSQL 内核本身一样 —— 普及到每一个用户手中，而不是必须向赛博空间上的封建云领主花大价钱租赁。

所以我打造了 Pigsty —— 一个开箱即用的开源 PostgreSQL 数据库发行版，旨在凝聚 PostgreSQL 生态扩展的合力，并把提供优质数据库服务的能力普及到每个用户手中。

Pigsty 是 PostgreSQL in Great STYle 的缩写，意为 PostgreSQL 的全盛状态。我们提出了六点核心价值主张，对应 PostgreSQL 数据库服务中的的六个核心问题：Postgres 的可扩展性，基础设施的可靠性，图形化的可观测性，服务的可用性，工具的可维护性，以及扩展模块和三方组件可组合性。

Pigsty 六点价值主张的首字母合起来，则为 Pigsty 提供了另外一种缩写解释：

Postgres, Infras, Graphics, Service, Toolbox, Yours.

属于你的图形化 Postgres 基础设施服务工具箱。

可扩展的 PostgreSQL 是这个发行版中最重要的价值主张。我们希望整合 PostgreSQL 生态里的各种力量，并将其凝聚在一起形成合力，打造一个数据库世界中的 Ubuntu 发行版。而我相信，这里才会是数据库世界下一阶段的真正竞争焦点所在！

在刚刚发布的 Pigsty v2.6 中，我们整合了上面提到的 DuckdbFDW 与 ParadeDB ，这两个插件让 PostgreSQL 的分析能力得到史诗级增强，而我们确保每个用户都能轻松用得上这样的能力。
